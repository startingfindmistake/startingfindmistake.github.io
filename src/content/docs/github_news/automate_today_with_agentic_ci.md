일단 CI에 대해 간단하게 알아보고 내용을 확인해보면
> **지속적 통합(Continuous Integration)**
여러 개발자가 작성한 코드(변경 사항)를 하루에도 여러 번 중앙 저장소에 합치고, 자동으로 빌드 및 테스트하는 방식입니다.
>
> * 이것을 통해 얻을 수 있는 효과는 코드 통합 문제를 초기에 발견하여 품질을 높이고, 개발 속도를 향상 시킵니다.
>
> 	CI/CD: 지속적 통합(CI)와 지속적 배포(Continuous Deployment)를 묶어서 부르는 말로, 현대 소프트웨어 개발의 자동화 흐름을 의미합니다.
>
>      * CI서버: GitHub Actions, GitLab CI, Jenkins 등 자동화된 빌드/테스트를 수행하는 도구가 있습니다.



CI는 정확성을 명확하게 표현할 수 있을 때, 즉 테스트 통과 또는 실패, 빌드 성공 또는 실패, 규칙 위반 또는 위반되지 않음과 같은 경우에 탁월한 성능을 발휘합니다.

하지만 CI는 의도적으로 휴리스틱과 규칙으로 환원(원래의 상태)될 수 있는 문제로 제한됩니다.

<pre>
휴리스틱(Heuristics): 시간이나 정보가 제한적일 때, 복잡한 문제를 완벽한 논리 대신 직관, 경험, 어림짐작을 통해 빠르게 판단하고 해결하는 방식
</pre>


대부분 팀에서는 작업 코드를 작성하는 것보다 코드와 관련된 변경 사항 검토, 문서 정확성 유지, 의존성 관리, 회귀 추적, 테스트 유지 관리, 품질 모니터링, 그리고 코드 배포 후에야 드러나는 문제에 대응하는 작업입니다.

하지만 엔지니어링은 작업의 상당 부분은 확정적인 검증보다는 해석, 종합, 맥락 파악을 필요로 한다.
그리고 엔지니어링은 작업중 점점 더 많은 부분이 CI가 담당하는 범주내 역활과 다르게 코드 의도 이해에 의존하는 작업에 속하게 됩니다.

## CI만으로는 더 이상 충분하지 않은 이유

CI는 이진 결과(통과 또는 실패)를 위해 설계되었습니다.
명확하게 정의된 위반 사항을 표시하면서
테스트는 통과하거나 실패하고, 빌드는 성공하거나 실패합니다.
이는 규칙 기반 자동화에 적합 합니다.

하지만 엔지니어링에서 가장 어렵고 시간이 많이 걸리는 부분 중 상단수는 판단이 많이 필요하고 상황에 따라 달라집니다.

* 문서에 해당하는 단어와 나와 있는 내용과 실제 구현이 다른경우
* 텍스트는 접근성 검사를 통과했지만 여전히 사용자에게 혼란을 줄 수 있는 경우
* 종속성이 새로운 플래그를 추가하여 주요 버전 업데이트 없이 동작을 변경합니다.
* 정규 표현식이 루프 내에서 컴파일되면 미묘한 방식으로 성능이 저하됩니다.
* 사용자 인터페이스 동작 변경 사항은 제품과 상호 작용할 때만 표시되는 경우
