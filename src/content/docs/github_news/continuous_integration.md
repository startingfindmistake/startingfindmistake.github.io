## Step1. CI(Continuous Integration) 지속석 통합이란?
---
GitHub에서는 CI를 개발자가 코드를 공유 Repository(저장소)에 자주 커밋(Commit)하도록 요구하는 소프트웨어 개발 방식으로 정의합니다.

단순히 코드를 자주 올리는 것에서 끝나는 것이 아니라, 새로운 코드가 리포지토리에 푸시(Push)될 때마다 CI 서버가 이를 감지하여 **자동으로 코드를 빌드(Build) 하고 테스트(Test)** 하는 전체 과정을 의미합니다.
이를 통해 새로 작성한 코드가 기존 코드 베이스에 통합될 때 오류를 일으키지 않는지 지속적으로 검증하게 됩니다.

## Step2. CI(Continuous Integration) 프로세스는 어떻게 작동하나요?
---
개발자가 Repository에 코드를 커밋하면, 다음과 같은 자동화된 검증 단계가 백그라운드에서 실행됩니다.
* 코드 스타일 검사(Linters): 팀에서 정한 코딩 컨벤션이나 포맷에 맞게 코드가 작성되었는지 확인합니다.
* 보안 검사(Security Checks): 코드에 알려진 보안 취약점이 포함되어 있지 않은지 스캔합니다.
* 테스트 실행(Functional Tests & Code Coverage): 단위 테스트(Unit Test)나 통합 테스트 등을 실행하여 애플리케이션의 기능이 의도한 대로 정상 작동하는지 확인합니다.


만약 이 모든 과정 중 하나라도 실패하면 CI(Continuous Integration)시스템이 즉각적으로 오류를 보고하여, 문제가 있는 코드가 메인 브랜치(Main branch)에 병합되는 것을 방지합니다.

## Step3. GitHub가 강조하는 CI의 도입 장점
---
* 조기 버그 발견 및 디버깅 시간 단축: 코드를 조금씩 자주 커밋하기 때문에, 테스트가 실패하더라도 방금 올린 적은 양의 코드 안에서만 문제의 원인을 찾으면 됩니다. 이는 디버깅에 소요되는 시간을 획기적으로 줄여 줍니다.
* 병합 충돌(Merge Conflict)최소화: 여러 명의 개발자가 각자 오랜 시간 작업한 뒤 한 번에 코드를 합치면 거대한 충돌이 발생하기 쉽습니다.
CI를 통해 코드를 자주 업데이트하고 통합하면 팀원 간의 변경 사항을 병합하기가 훨씬 수월해집니다.
* 개발 생산성 향상: 코드 검증과 테스트가 자동화되므로, 개발자는 버그를 찾고 충돌을 해결하는 데 시간을 낭비하는 대신 '실제 코드를 작성하는 일'에 더 많은 시간을 집중할 수 있습니다.
* 안정적인 코드 퀄리티 유지: 모든 커밋이 린터와 테스트를 통과해야만 병합될 수 있으므로, 항상 배포 가능한 수준의 안정적인 프로젝트 상태를 유지할수 있습니다.
