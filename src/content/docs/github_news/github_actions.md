title: 배포할때마다 Github actions하는데 뭘 알고 하는거야?
description: Github actions에 대해서 단계별로 자세히 알아보자
---

GitHub Actions는 특정한 "사건(Event)"가 발생하면, 정해진 "작업(Job)"들을 "서버(Runner)"에서 실행하는 구조로 되어 있습니다.


1. 워크플로우(Workflows)
* 리포지토리에 추가할 수 있는 자동화된 전체 프로세스 입니다.
* 하나 이상의 작업을 실행하며, `github/workflows` 디렉터리에 YAML파일 형식으로 작성됩니다.
* ex: 코드가 Push될 때마다 테스트하고 배포하는 워크플로우"

2. 이벤트(Events)
* 워크플로우 실행을 트리거(작동)하는 리포지토리 내의 특정 활동입니다.
* ex: 누군가 코드를 `push` 했을 때, Pull Request(PR)를 생성했을 때, 혹은 매일 밤 12시 같은 특정 시간(Cron 스케줄)에 발생할 수 있습니다.

3. 작업(Jobs)
* 동일한 러너(Runner)에서 실행되는 워크플로우 내의 일련의 단계(Steps)들입니다.
* 기본적으로 여러 Job은 병렬로 실행되지만, 의존성을 설정하여 순차적으로 실행하게 할 수도 있습니다.(ex: '빌드 Job'이 성공해야만 '배포 Job'이 실행되도록 설정)

4. 단계(Steps)
* Job 내에서 순차적으로 실행되는 개별 태스크 입니다.
* 운영체제의 쉘(Shell)명령어를 실행하거나, 미리 만들어진 'Action'을 실행할 수 있습니다.

5. 액션(Actions)
* GitHub Actions 플랫폼의 가장 강력한 특징이자 이름의 유래입니다.
* 자주 반복되는 복잡한 작업을 캡슐화한 독립적인 명령 단위 입니다.
*직접 만들 수도 있고, GitHub Marketplace에서 다른 사람이나 기업(ex: AWS, Google등)이 만들어 둔 수만 개의 Acion을 가져와 레고 블록처럼 조립해서 쓸 수 있습니다.

6. 러너(Runners)
* 워크플로우(정확히는 Job)가 실제로 실행되는 서버 컴퓨터 입니다.
* GitHub에서 Ubuntu Linux, Windows, macOS 환경이 세팅된 가상 머신(GitHub-hosted runner)을 기본적으로 제공하며, 필요하다면 내 개인 서버(Self-hosted runner)를 연결해서 사용할수도 있습니다.




## Step2. GitHub Actions로 뭘 할수 있는데 그래서?
---
단순히 코드 테스트(CI)뿐만 아니라
* CI/CD 파이프라인 구축: 코드가 메인 브랜치에 병합되면 자동으로 빌드하고, AWS나 Azure 같은 글라우드 서버에 배포(CD) 합니다.
* 이슈 및 PR 관리 자동화: 누군가 새 이슈를 열면 자동으로 특정 라벨(Label)을 달아주거나, 담당자에게 할당합니다.
* 코드 품질 검사: PR이 올라오면 자동으로 코딩 컨벤션 검사(Lint)를 수행하고, 위반 사항이 있으면 PR에 자동으로 코멘트를 남깁니다.
* 정기적인 작업 실행: 매주 월요일 아침에 프로젝트 진행 상황 리포트를 슬랙(Slack)으로 자동 발송합니다.
